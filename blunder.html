Overview

Use consistent and meaningful error codes that indicate the type and severity of the error. For example, you can follow the HTTP status code convention, which uses different ranges of codes for different categories of errors, such as 4xx for client errors and 5xx for server errors. You can also use custom codes or subcodes to provide more specific information about the error, such as 400.1 for invalid parameters or 500.2 for database connection failure. However, make sure that your error codes are well documented and standardized across your API endpoints and versions. We use error response standard by google/mastercard/ibm.

Cause

Provide clear and helpful error messages that explain the cause and solution of the error. For example, you can use the default JSON format to return a structured error response that contains the error code, message, details, and suggestions. You can also use plain language and avoid technical jargon or abbreviations that may confuse the user or developer. Additionally, you can use localization and internationalization techniques to provide error messages in different languages and formats according to the user's preferences and location.

Logging

We support central error logging and global error handling that can help you track, analyze, and resolve errors in your API.

Guides

We are following the fail-fast and fail-safe principles that can help you prevent and recover from errors in your API. The fail-fast principle means that you should validate and verify your API inputs, outputs, and expectations as early and as often as possible, and stop the test execution if any error occurs. This can help you avoid wasting time and resources on testing invalid or incorrect scenarios, and identify and fix the errors quickly. The fail-safe principle means that you should design and implement your API tests with fallback mechanisms and recovery strategies that can handle and mitigate the impact of any error. This can help you ensure the continuity and reliability of your API testing, and avoid catastrophic failures or data loss.
